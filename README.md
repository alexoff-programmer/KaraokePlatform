# Караоке Платформа (Конвертер MP3 в MP4 с ИИ-субтитрами)

Веб-приложение для автоматической генерации караоке-видеороликов на базе .NET 10 (Razor Pages + SignalR), Python-утилиты `audio-separator` (для ИИ-изоляции вокала) и нейросети Whisper (для посимвольного распознавания текста).

Проект полностью контейнеризирован с помощью **Docker**. Все тяжелые зависимости (FFmpeg, ffprobe, Python 3, библиотеки компиляции C-расширений) разворачиваются автоматически внутри изолированного окружения и не требуют настройки на хост-системе, что исключает конфликты с другими сервисами на сервере.

---

## 🚀 Быстрый запуск (Docker Compose)

Чтобы развернуть проект на сервере, перейдите в корень папки проекта (где находятся файлы `docker-compose.yml` и `Dockerfile`) и выполните команду:

```bash
docker-compose up --build -d

```

### Что произойдет при первом запуске:

1. Docker изолированно скомпилирует .NET 10 приложение (`Release`).
2. Внутри рантайм-контейнера развернется окружение Python 3, скомпилируются и установятся пакеты `audio-separator` и утилита `ffmpeg`.
3. На хост-машине автоматически создадутся постоянные папки для сохранения данных (`instance`, `Models`, `wwwroot/uploads`, `wwwroot/output`).
4. При первой обработке трека приложение самостоятельно скачает необходимые ИИ-модели вокала и текста в примонтированные папки. При последующих перезапусках контейнера скачивание происходить не будет.

## 💾 Постоянные данные и тома (Docker Volumes)

Чтобы база данных, загруженные пользователями медиафайлы и скачанные ИИ-модели весом в несколько гигабайт не стирались при обновлении или перезапуске контейнера, в `docker-compose.yml` настроен проброс папок (Volumes).

На сервере в папке проекта всегда будут находиться:

* `./instance/` — папка с базой данных SQLite (`karaoke.db`).
* `./Models/` — папка с тяжелыми бинарными моделями распознавания речи Whisper (например, `ggml-medium.bin`).
* `./Fonts/` — системные конфигурации и файлы кастомных шрифтов (например, `Montserrat-Bold.ttf`) для корректного рендеринга текста на видео через FFmpeg.
* `./wwwroot/uploads/` — исходные аудиофайлы (`.mp3`) и фоновые изображения, загруженные пользователями через веб-интерфейс.
* `./wwwroot/output/` — готовые выходные видеоролики караоке (`.mp4`) и файлы разметки субтитров (`.ass`).

## 🌐 Настройка Nginx (Reverse Proxy)

Приложение внутри Docker-контейнера слушает порт `8080`, который проброшен наружу на порт **`5000`** хост-машины (проект доступен локально на сервере по адресу `http://localhost:5000`).

Чтобы связать караоке-платформу с внешним доменом или IP-адресом и обеспечить стабильную работу прогресс-бара генерации видео (SignalR), добавьте в конфигурационный файл Nginx следующую настройку.

### 1. Добавление глобальной карты WebSockets

Убедитесь, что этот блок конфигурации находится **ВНЕ** блока `server {}` (например, в самом начале файла конфигурации сайта или в `nginx.conf`), он необходим, чтобы Nginx не разрывал постоянные WebSocket-соединения SignalR:

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

```

### 2. Конфигурация виртуального хоста `server`

```nginx
server {
    listen 80;
    server_name YOUR_DOMAIN_OR_SERVER_IP; # Укажите домен сайта или IP сервера

    # Критично для загрузки тяжелых аудиофайлов и фоновых изображений
    client_max_body_size 150M;

    # Локация 1: Проксирование основного веб-интерфейса (Razor Pages, Static Files)
    location / {
        proxy_pass http://localhost:5000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection keep-alive;
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Локация 2: Проксирование SignalR WebSockets (трансляция прогресса ИИ в реальном времени)
    location /notificationHub {
        proxy_pass http://localhost:5000;
        proxy_http_version 1.1;
        
        # Настройка заголовков для удержания постоянного WebSocket-туннеля
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 3600s; # Увеличиваем таймаут для длительных сессий рендеринга
    }
}

```

### 3. Применение изменений в Nginx

После сохранения файла конфигурации выполните валидацию синтаксиса и перезапустите веб-сервер Nginx:

```bash
sudo nginx -t
sudo systemctl restart nginx

```
